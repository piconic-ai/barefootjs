`augmentInheritedPropAccesses` (the shared pass every template-stash adapter runs) now sees `props.X` reads carried by dynamic text children (`<p>{props.label}</p>`), conditional / if-statement conditions (`{props.show && …}`), loop array expressions (`(props.items ?? []).map(…)`), and child-component prop values (`<Child value={props.foo} />`). Previously these reads were invisible for an untyped `props` parameter, so the emitted SSR template referenced a variable the generated props type / manifest `ssrDefaults` never declared — a strict-mode 500 on the Perl-family adapters and a missing struct field on Go (#2126 follow-up). A read of the form `props.X ?? <literal>` now also pins the synthetic param to the literal's primitive type (previously the string default made Go's prop-derived signal seeding emit `Count: in.Initial` with mismatched types); other text reads keep the `string` default, condition / loop-array / component-prop reads land in the nillable bucket, and boolean-HTML-attr name inference does not apply to component props. A new cross-adapter conformance fixture (`untyped-props-reads`) renders the text / condition / loop carriers with no props on every adapter against the Hono reference; the component-prop carrier and type-pinning rules are pinned by unit tests.
